The Solution

A fraud layer the bank runs, invisible until it isn't

Not a bank. A protection layer that fuses trust scoring, scam-message detection, and honeytoken breach alerts into one risk signal, without changing how customers already bank.

Three escalating response tiers

1

Soft Step-Up Medium risk

An action-bound one-time code, delivered the way a real bank text arrives: to the user's messages, never leaked back through the app.

2

Hard Step-Up High risk

On-device facial verification, confirming this is really the account holder.

3

Liveness Check Last resort

A guided facial check (center face, turn head, open mouth) gives a genuine user one more chance, instead of a dead-end block.

Two silent layers

NLP scam-message detection

Screens SMS at the device level, not inside the bank app, flagging phishing and smishing before a customer taps a bad link.

Honeytoken breach detection

Decoy account fields no legitimate user would ever touch. Any access is definitionally an intrusion, triggering an immediate freeze.

For the bank: a live security-ops dashboard

Active sessions, risk-tier counts, and a real-time event feed — visibility a static rules engine can't give.

Tech Stack

Built for a live demo, architected for production

⚛️

Frontend

React, Next.js 16 (App Router)

Server components by default, fast iteration, real-time UI updates.

🖥️

Backend

Node.js, Express

A lightweight REST API, easy to reason about for a judged demo.

⚡

Real-time

Socket.IO

Pushes trust-score events, scam scans, and breach alerts live — no polling.

🗄️

Database

PostgreSQL, Neon serverless

Relational integrity for accounts, transactions, decoy fields. Scales to zero between runs.

🧠

NLP

Google Gemini (gemini-2.5-flash)

Structured classification of inbound messages into legitimate/suspicious, with per-literacy alert copy.

🔑

Auth

JWT

Stateless session tokens, no server-side session store needed.

ℹ️ NLP note: scam classification was designed for the Claude API. We switched to Gemini during the build due to credit constraints. The prompt, schema, and tests are model-agnostic, so swapping back is a same-shape change, not a rebuild.

How it all connects

SentinelIQ Backend

Node.js & Express. Routes: /auth · /trust-score · /scan-message · /otp/*

PostgreSQL (Neon)

accounts · txns · decoy_fields · trust_events

Google Gemini API

scam-message NLP classification

Simulated SMS inbox

OTP delivery target — demo-grade, no real gateway yet

Flow: a transfer is initiated → scored on device, location, velocity, and recipient risk → handled by proceed, OTP, facial verification, or a liveness check. Every event streams live to the admin dashboard; a touched decoy field freezes the account instantly, independent of the score.

What Comes Next

From demo to deployment

💬

Real SMS gateway

Swap the simulated inbox for a carrier gateway (Termii, Africa's Talking) so OTPs arrive as real texts.

🙂

Production biometrics

Replace simulated facial & liveness with a certified on-device SDK (native APIs or FaceTec) for spoof resistance.

🧠

Claude API migration

Move the NLP classifier from Gemini back to Claude as designed. The prompt, schema, and tests are already model-agnostic.

🌐

Multi-language detection

Extend scam classification to Pidgin, Yoruba, Hausa, and Igbo, matching how scam SMS actually arrives.

🏦

Bank-side rollout

Package the trust engine & dashboard as an integration fraud teams adopt without replacing core banking.

✅

Regulatory alignment

Pursue CBN & NDPR review, positioning for the liveness and device-binding mandates already arriving.

Also on the roadmap: mule-account detection, cross-bank recipient risk scoring, and behavioral biometrics (typing rhythm, session patterns) beyond today's device, location, and velocity signals.
